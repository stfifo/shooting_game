### 1. 평가 목적

본 보고서는 Strikers 1945+ 스타일의 스크롤 슈팅 게임 개발에서 Claude Code가 생성한 코드의 품질을 두 가지 방법으로 평가하고, 사람-기계 평가 일치도를 측정한 결과를 담았다.

- **평가 방법 1 (M1)**: 3차원 다차원 Rubric — Correctness/Completeness/Visual Design 각 1~5점으로 평가한 후 평균으로 점수를 산출하였다.
- **평가 방법 2 (M2)**: 종합 Rubric — 케이스 전체를 1~5점으로 단일 점수로 평가하였다.
- **일치도 지표**: Cohen's Kappa (`sklearn.metrics.cohen_kappa_score`), `weights=None` / `weights='quadratic'` 두 가지로 진행하였다.


### 2. 평가 방식 및 선택 이유

자세한 평가 기준은 eval 폴더의 `rubric.md`에 나와있다.

#### 방식 1 : 다차원 Rubric (M1)

| 차원 | 의미 |
|---|---|
| Correctness | 구현이 버그 없이 의도대로 동작하는가 |
| Completeness | 요구사항 전체를 빠짐없이 구현했는가 |
| Visual Design | 시각 효과/HUD/애니메이션이 게임 스타일에 맞게 구현되었는가 |

게임에서는 '동작 정확도', '기능 완성도', '시각 표현'이 중요하기 때문에 개별적으로 평가하였다. 차원별 분리 평가로 부분적 성공과 실패를 세밀하게 포착할 수 있었다.

#### 방식 2 : 종합 Rubric (M2)

- 게임 전체를 하나의 점수로 평가. 1~5점 각각에 명확한 기준을 명시(edge 케이스. UI 가독성, 클리어 가능 여부 등)하여 평가자 간 기준 편차를 줄였다.
- 선택 이유: 다차원 평가는 차원 간 가중치 결정이 필요하지만, Rubric은 전체적 완성도를 직관적으로 표현한다. 두 방식을 함께 사용함으로써 세부 진단과 전체 품질 평가를 모두 수행한다.


### 3. 사용 라이브러리

#### Claude Code CLI (LLM-as-Judge)

LLM 자동 평가는 외부 API 호출 없이 **Claude Code 세션 내에서 직접** 수행했다. Claude Code의 fork agent 메커니즘을 활용하여, 각 테스트케이스 백업 폴더의 핵심 JS 파일을 Read 도구로 직접 읽고 rubric 기준에 따라 코드를 정적 분석하여 점수(correctness, completeness, visual_design, m2)를 산출한 뒤 `eval/llm_scores.json`에 저장했다. `anthropic` Python SDK나 별도 API 키는 사용하지 않았다.

#### scikit-learn (`sklearn`)

```python
from sklearn.metrics import cohen_kappa_score
kappa_unweighted = cohen_kappa_score(human_scores, llm_scores, weights=None)
kappa_quadratic  = cohen_kappa_score(human_scores, llm_scores, weights='quadratic')
```

사람 점수(person_eval.md)와 LLM 점수(llm_scores.json)를 리스트로 변환하여 `cohen_kappa_score`에 전달했다.

### 6. weights 유무에 따른 차이 설명

- `weights=None`  
    unweighted Kappa는 점수 차이의 크기를 무시한다. 1점 차이든 4점 차이든 동등하게 '불일치'로 처리함을 볼 수 있다
- `weights='quadratic'`  
    가중 Kappa는 불일치 정도에 이차 가중치를 적용한다. 1점 차이는 가볍게, 4점 차이는 16배 무겁게 패널티를 준다. 5점 척도처럼 순서가 있는 데이터에서 '큰 불일치가 드물다'면 unweighted보다 높은 값이 나오는 경향이 있다.

> 본 평가에서는 M2 기준 unweighted=0.0244, quadratic=0.3421로, quadratic이 더 높게 나왔다. 이는 사람과 LLM의 큰 의견 불일치(3점 이상 차이)가 적다는 의미로 볼 수 있다


### 7. 사람-기계 일치도 분석

전체 10개 케이스에 대해 M2 종합 점수 기준 Cohen's Kappa는 unweighted=0.0244, quadratic=0.3421이다.

- LLM 평가의 특성
  - LLM(claude-sonnet-4-6)은 코드 정적 분석 기반으로 평가했다. 실제 브라우저 실행을 통한 동적 테스트 없이, 코드 로직과 함수 구조, 패턴 구현 여부 등으로 점수를 산출했다.
  - 이로 인해 시각적인 디자인 차원에서 LLM은 Canvas 렌더링 코드의 존재 여부만 판단하며, 실제 화면에서의 자연스러움과 가독성은 반영되지 않는다.
  - Correctness 차원에서 LLM은 확실한 버그(변수 미초기화, 조건 오류 등)는 잘 잡아내지만, 미묘한 타이밍 이슈나 브라우저 특정 버그는 놓칠 수 있다.
- **일치도 해석**  
    Kappa가 0.4 미만으로 일치도가 낮다. LLM 정적 분석과 사람의 실행 기반 평가 간에 체계적 차이가 존재할 가능성이 높다. LLM은 코드가 존재하느냐에 집중하는 반면, 사람은 동작의 자연스러움과 게임 플레이 경험까지 반영하였기 때문에 이런 차이가 보인다고 생각한다. 특히 시각 디자인과 난이도 체감처럼 주관적 요소가 강한 항목일수록 사람과 LLM 평가자 간의 괴리가 커질 것으로 예상된다.
